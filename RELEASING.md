# Выпуск версии

## Когда создавать релиз

Создавайте релиз после изменений, проверенных GitHub Actions и на чистом VPS. Для первого публичного выпуска рекомендуется `v0.1.0`.

## Нумерация

- `v0.1.1` — исправления без новых возможностей.
- `v0.2.0` — новые возможности без изменения существующего сценария.
- `v1.0.0` — первая стабильная версия после нескольких успешных реальных установок.
- Следующие major-версии — изменения, требующие другого порядка установки или несовместимые с прежними настройками.

## Перед выпуском

- [ ] Проверить Pull Request: Bash syntax и ShellCheck.
- [ ] Установить standalone-сервер на чистом Ubuntu VPS с доменом.
- [ ] Проверить удалённую ноду.
- [ ] Проверить WARP: включение, отключение и маршрут к российским доменам/IP.
- [ ] Проверить просмотр настроек, восстановление и полное удаление.
- [ ] Обновить [CHANGELOG.md](CHANGELOG.md).
- [ ] Убедиться, что в коммитах и файлах нет секретов.

## Публикация

1. Создайте Draft Release в GitHub.
2. Создайте тег, например `v0.1.0`, на проверенном коммите `main`.
3. Добавьте краткие release notes: новые возможности, исправления и известные ограничения.
4. Опубликуйте релиз только после финальной проверки текста и файлов.

GitHub Release фиксирует конкретный тег и позволяет пользователям установить проверенную версию, а `main` оставить для актуальной разработки.
